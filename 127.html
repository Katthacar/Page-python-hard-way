This technique is very useful when you need to write a replacement for a terrible piece of software. A
common job in software is to work on a project that aims to replace an older system with a more modern
implementation. An example is replacing an old COBOL banking system with a fresh, new, hot Django
system. The motivation is typically to make it easier to maintain and extend by using something easier
to work with than the old system. If you can write a set of automated tests that validate the behavior of
the old system and then point that test suite at the new system, then you have a way to conﬁrm that
your replacement works
...
mostly. Trust me, these replacement jobs are nearly impossible and don’t
succeed too often, but an automated test helps.
